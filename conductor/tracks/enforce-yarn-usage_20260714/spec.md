# Specification: Enforce Yarn as Package Manager

**Track ID:** enforce-yarn-usage_20260714
**Type:** Chore
**Created:** 2026-07-14
**Status:** Draft

## Summary

Migrate the ClinicCare project from npm to Yarn as the official package manager. This includes generating a `yarn.lock`, adding a `packageManager` engine constraint, configuring CI/CD workflows, and updating all developer-facing documentation to reflect Yarn commands.

## Context

ClinicCare is a React 19 + Vite + TypeScript SPA with a Firebase backend. The project currently uses npm (evidenced by the absence of `yarn.lock` and the use of `npm ci` / `npm run *` throughout CI/CD pipelines). Standardizing on Yarn improves install reproducibility, enables workspaces if the project grows to a monorepo, and aligns with team preference.

## Motivation

- Remove ambiguity about which package manager to use (no lock-file enforcement today).
- Gain faster, deterministic installs via `yarn install --frozen-lockfile` in CI.
- Prepare the toolchain for possible future monorepo adoption.

## Acceptance Criteria

- [ ] A `yarn.lock` file is committed to the repository root.
- [ ] `package.json` includes `"packageManager": "yarn@<version>"` and an `"engines": { "yarn": ">=1.22.0" }` field.
- [ ] A `.npmrc` or `.yarnrc` file disallows accidental `npm install` (`engine-strict=true` or `yarnPath` configured).
- [ ] All four GitHub Actions workflows (`ci.yml`, `cd.yml`, `firebase-hosting-merge.yml`, `firebase-hosting-pull-request.yml`) use `yarn install --frozen-lockfile` and `yarn <script>` instead of `npm ci` / `npm run`.
- [ ] The `deploy` script in `package.json` is updated from `npm run build` to `yarn build`.
- [ ] `doc/guias/Technical_Roadmap_and_Setup.md` is updated: all `npm install`, `npm run *`, and `npm test` commands are replaced with their Yarn equivalents.
- [ ] Local installation succeeds and `yarn dev`, `yarn test`, `yarn build` all pass.

## Dependencies

None — self-contained toolchain chore with no dependency on other tracks.

## Out of Scope

- Migrating to Yarn PnP (Plug'n'Play) or Yarn workspaces — classic mode only.
- Adding or removing any production or dev dependencies.
- Making any application code changes.

## Technical Notes

- Use **Yarn Classic (v1.x)** for maximum compatibility with the existing Vite/Storybook/Firebase toolchain unless the team explicitly wants Berry (v4).
- On the CI actions that use `actions/setup-node@v4`, switch the `cache:` field from `'npm'` to `'yarn'`.
- The auto-generated Firebase Hosting workflows (`firebase-hosting-*.yml`) must be manually edited because Firebase CLI re-generates them on re-init; add a comment warning developers not to overwrite them.
